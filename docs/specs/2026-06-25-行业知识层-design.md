# 行业知识层（Vertical Knowledge Layer）— 设计 spec

- 日期：2026-06-25
- 状态：已通过 brainstorming，待出实现计划
- 所属 skill：pingcode-presales
- 关联：本 spec 为子项目 **A**；执行层 **B（自定义类型/字段可靠重放）** 单独立项

## 1. 背景与动机

当前 skill 搭**通用 WBS** 没问题，但**垂直行业（汽车电子、医疗器械等）的结构搭不对**，卡点集中在「自定义工作项类型 + 字段属性」。根因经确认是**两头都缺**：

- **知识缺**：不知道某行业该有哪些专属类型/字段/阶段门/术语（"建什么"）。
- **执行缺**：即便知道，PingCode 后台建自定义类型/字段那步（path③ 的 3 层 CDK 弹窗）脆弱/易断（"怎么建"）。

本 spec 只解决**知识缺（A）**。执行缺（B）单独立项；A 产出的「类型+字段」清单即 B 的输入。A 不依赖 B 也能先交付价值（其余结构今天就能 API 搭，类型/字段清单进蓝图当 spec 用）。

## 2. 目标与非目标

**目标**
- 给搭建流程装一层"行业大脑"：按客户行业，产出**完整行业结构包**塑造蓝图。
- 大行业精 curated、长尾现场调研生成、生成物经认可可沉淀为 curated（混合，越用越厚）。

**非目标（明确排除）**
- ❌ B：可靠的 Web 自定义类型/字段重放/录制（另立项）。
- ❌ 替换现有通用 WBS 逻辑——A 是**在通用基线上叠加行业 delta**，无模板时降级回通用。

## 3. 已定决策（brainstorming 结论）

| 维度 | 决策 |
|------|------|
| 覆盖面 | **混合**：curated 大行业 + 长尾现场生成 + 生成物可沉淀 |
| 模板范围 | **完整行业结构包**：项目类型 / 行业阶段门流程 / 专属类型+字段 / 测试维度 / 合规术语 |
| 模板来源 | **我起草 baseline（调研公开标准）+ 你叠实战增量** |
| 存储格式 | **混合**：Markdown 散文区（LLM 读/人编辑）+ 结构化 YAML 区（执行层精确消费） |

## 4. 架构

- 新增目录 `references/verticals/`，**一个行业一个文件**（`汽车电子.md`、`医疗器械.md`…）。
- 每文件 = 该行业完整结构包。
- **消费方式**：绝大部分由 LLM 在 Phase 2→3 阅读以塑造蓝图；唯独 YAML 的 `types` 块**精确交给执行层 / 建类型逻辑**。
- 这是把现有 `scripts/pingcode_web.js` 里扁平硬编码的 `WORK_ITEM_TYPE_CONFIGS`（6 类型）**升级为按行业组织**。

## 5. 文件格式

每个 `references/verticals/<行业>.md`：

```markdown
# 汽车电子 — 行业结构模板

## 概览  （散文区：人编辑，LLM 读）
- 项目类型: hybrid（ASPICE V 模型）
- 阶段门: 概念 → 系统需求 → 架构设计 → 详细设计 → 实现 → 集成测试 → 系统测试 → 验收
- 工作项层级要点: 系统需求/软件需求双轨 + 需求-测试追溯
- 测试维度: HIL/SIL、EMC、功能安全(ISO 26262)、可靠性
- 标准: ASPICE / ISO 26262 / IATF 16949
- 术语: ECU, T-Box, ASIL, DFMEA, AUTOSAR, OTA
- 客户关注点(实战增量): ⟵ 你叠加

## 专属类型与字段  （结构化区：执行层精确消费）
​```yaml
types:
  - name: 工程变更申请
    group: 事务            # 仅限 需求 / 任务 / 事务
    fields:
      - { name: 变更类型, kind: 单选, options: [设计变更, 工艺变更, 材料变更, 软件变更] }
      - { name: 影响分析, kind: 多行文本 }
  - name: 功能安全需求
    group: 需求
    fields:
      - { name: ASIL等级, kind: 单选, options: [QM, A, B, C, D] }
      - { name: 安全目标, kind: 多行文本 }
​```
```

**格式约束**
- 散文区无 schema 负担，自由编辑。
- YAML `types[].fields[].kind` **必须**落在 PingCode 13 种属性类型内（单行文本/多行文本/单选/多选/数字/日期/成员/级联单选/级联多选/进度/评分/链接/引用）。
- `group` 仅限 `需求 / 任务 / 事务`。
- 单选/多选必须带 `options`。

## 6. 匹配 → 生成 → 沉淀 闭环

Phase 2 内新增「选行业模板」步：

```
行业文本 → 在 references/verticals/ 匹配（LLM 按别名/语义就近）
  ├─ 命中 curated         → 直接加载
  ├─ 没命中（长尾）        → 现场 WebSearch 行业流程/标准/术语 → 生成内存版结构包（同 md+yaml 形态）
  └─ 都不沾边 / 调研薄     → 降级走通用 WBS（=今天）+ 标注"本行业未沉淀模板"
```

**沉淀回流**：交付时（Phase 6），若本次为现场生成的长尾行业，提示"是否沉淀成正式模板？"。认可 → 写 `references/verticals/<行业>.md`（你过目/微调）→ 提交。下次同行业即从"长尾生成"升级为"curated 命中"。

## 7. 集成进 skill 流程

- **挂载点**：Phase 2（调研）末，产出"行业结构包"。
- **塑造蓝图**：包内容驱动 Phase 3.1（项目/阶段门规划）、命名术语、测试维度；`types` 块流向 Phase 3.7 / B。
- **确认**：行业结构随整份蓝图在 Phase 3.6 一并确认（不新增确认关）。
- **与 B 接口**：`types` YAML 块即 B 的输入清单。B 未就绪前，该块照进蓝图与交付摘要当 spec（手工建或走现有 path③）；B 就绪后无缝对接，A 不改。

## 8. 起草 / 维护工作流

- 我调研标准 + 行业流程，起草 curated 初稿 → 提交。
- 你直接编 md 散文区「客户关注点」叠实战增量。
- 长尾生成物经你认可沉淀为 curated。
- 维护 = 改文件 + 提交 GitHub，同事 `git pull` 即得（复用现有发布链路）。

**Baseline 来源优先级**：PingCode 官方行业方案 / 能力匹配分析（如有）> 公开标准研究 > 通用推断。官方材料的定位/术语/案例客户最认（如汽车电子已有《汽车电子行业研发管理解决方案》《面向 ASPICE 的 PingCode 能力匹配分析》）。

**专属类型优先补 PingCode 短板（重要原则）**：用行业标准对照 PingCode 能力，**缺口处正是最该建的自定义类型**——例：ASPICE `SUP.10 变更请求管理` PingCode 标注不满足 → 建「变更请求」类型；强追溯/基线/覆盖率短板 → 建「基线评审」「追溯关联」相关类型。这样自定义类型既补功能短板、又是演示时讲合规故事的抓手，比"行业有这些类型"的理由更硬。

## 9. 容错 / 降级

| 情况 | 处理 |
|------|------|
| 无模板且调研薄 | 走通用 WBS，标注建议补模板 |
| `kind` 非法属性类型 | 起草/生成时即校验，非 13 种合法属性不放行 |
| 行业标准冲突（功能安全 vs 敏捷） | 进 Phase 3.6 让用户拍板 |
| 现场生成质量存疑 | 蓝图标注"⚠️ 现场推测"，复用 Phase 2 不确定标注机制 |

## 10. 验证

1. **机器层**：YAML 可解析；每个 `kind` 映射到 13 属性类型；`group` 合法；单/多选带 options。
2. **端到端**：拿 curated 汽车电子跑一次搭建，确认蓝图阶段门=ASPICE、类型字段=模板所列（非通用）。
3. **SME 关**：贴不贴行业由用户眼判——质量终审，机器只保证不崩。
