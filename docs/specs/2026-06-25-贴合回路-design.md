# 贴合回路（Fit-Loop）— 设计 spec

- 日期：2026-06-25
- 状态：已通过 brainstorming，待出实现计划
- 所属 skill：pingcode-presales
- 关联：依赖**行业知识层（Vertical Knowledge Layer）**产出"骨架"，但本 spec **不含**骨架生成，骨架为相邻在飞件。

## 1. 背景与动机（第一性原理）

售前搭的 POC，准确度的裁判只有一个：客户看着它信不信"**这就是我家、我们团队真能这么跑**"。把这个"准确"剥到底，得到三条递进结论：

1. **准确度 ≈ context 保真，不是虚拟数据。** fit（贴合度，"是不是他们"）由 调研→结构→术语 决定，是致命项；realism（拟真度，"像不像真在干活"）才由虚拟数据决定，是二阶。骨架错，再逼真的燃尽图也救不回；结构对，数据冷也能现场讲。虚拟数据唯一承重处是"度量/看板"那条能力线。

2. **context 有成色：先验（猜）vs 接地（证）。** 表单+网搜+行业模板 = 段位级**先验**（对行业大致对、对这个客户不一定）；客户亲口/原始文档 = 账户级**接地**。"这不是我们"本质就是 先验 ≠ 接地 的那道缝。

3. **客户不知道自己要什么。** 所以客户的**自我报告不是地面真相，是噪声**；可靠的精准 context 是他**不知道也改变不了**的东西——**身份**（行业/产品/规模/被强加的标准）+ **该身份的最佳实践**（=行业大脑）。于是：

> **fit = 「他的皮（真名真词，从他的料里抽）」套在「最佳实践骨架（行业大脑给）」上。** 太通用→"这不是我们"（皮没贴）；照搬现状→"这没价值"（骨架没给）。在两个死法间穿针，就是这活的本质。
>
> 而客户没法凭空说需求、只会**对着具体草案反应**——所以 POC 不是交付物，是**探针**；必须 `草案 → 反应 → 重渲染` 地 **loop**。本 spec 就是把这条 loop 做成机器。

## 2. 目标与非目标

**目标**
- 把 POC 从"一次性猜测"变成"随销售周期收敛到客户真实世界"的活资产。
- 用一条管道 + 两个输入适配器，把**接地 context**（客户文档 / 会谈反应）灌进客户模型，差分渲染到已建 PingCode 环境，dry-run 把关。

**非目标（明确排除）**
- ❌ **骨架生成 / 行业大脑**——相邻在飞件（`行业知识层` spec），本 spec 只**消费**骨架，不产骨架。
- ❌ **自动触发 / `/loop`·`/schedule` 无人值守马达**——人手转曲柄即可，自动化是以后可选层。
- ❌ **跨客户驾驶舱**、❌ **`.mpp`(MS Project) 解析**（实战很少，延后）。
- ❌ 重建已有搭建逻辑——本件是在 `build_manifest` 之上叠"增量修订"，首建仍走现有 Phase 4。

## 3. 已定决策（brainstorming 结论）

| 维度 | 决策 |
|------|------|
| 范围 | **回路 + 贴皮** 一起；骨架相邻件 |
| 架构 | **方案 1**：context store 做带 provenance 的补丁日志；贴皮/反应都是 writer；差分渲染 + dry-run 闸 |
| 合体洞察 | 贴皮与反应是**同一输出类型（context 补丁）的两个适配器**，后面共用 store→渲染→闸。"回路+贴皮"在代码里是**一条管道 + 两个输入适配器**，不是两套 |
| loop 形态 | **事件驱动、人手触发**（每来一份文档 / 每见一次客户跑一次），非定时；不引入 loop 基建 |
| 漂移处理 | **信任 manifest**（不上"每轮全量 GET 对账"的方案 3）；代价由 dry-run 预览兜 |

## 4. 架构

五个单元，各自单一职责、接口清晰、可独立测：

| 单元 | 做什么 | 依赖 |
|------|--------|------|
| **Context Store** `{客户}_context.json` | 客户模型唯一真相源：一串带 provenance+状态的 fact | 纯数据 |
| **Skin 适配器** | `文档/官网 → context 补丁`（贴皮） | `pingcode-wbs` 语义拆 |
| **Reaction 适配器** | `会谈纪要 → context 补丁`（反应） | `followup-extraction` 输出 |
| **Diff 渲染器** | `(store, manifest) → 计划{增/改/删}`，纯函数 | `build_manifest` |
| **Applier + dry-run 闸** | 打印计划→确认→执行→回写 manifest/状态 | `pingcode_api` |

**数据流**

```
表单/网搜/骨架(隔壁件) ─┐
文档 ───[Skin 适配器]───┤
会谈 ──[Reaction 适配器]┘──▶ context.json (facts: 内容 + 来源 + 状态)
                                   │
                   build_manifest ─┤
                                   ▼
                          Diff 渲染器 ──▶ 计划(N增 / M改 / K删)
                                   │  dry-run 预览
                                   ▼ (确认)
                          Applier ──▶ PingCode 环境 + 回写 manifest & 状态
```

**复用既有资产**：`build_manifest`（增量基准）、`followup-extraction`（反应源之一）、`pingcode-wbs`（贴皮源之一）、`vocab.json`（成为 store 的派生视图）。

## 5. Context Store 形状

一笔 fact = 一个原子 context 单元：

```json
{
  "client": "某车灯Tier1",
  "skeleton_ref": "汽车电子",
  "identity": { "industry": "汽车电子", "products": ["ADB自适应远光"], "scale": "50-200", "standards": ["ASPICE", "ISO 26262"] },
  "facts": [
    { "id": "f012", "path": "需求/软件需求/ADB分区点亮", "kind": "work_item",
      "value": "ADB分区点亮算法", "source": "doc:reqs.xlsx#row12", "status": "confirmed", "authority": null },
    { "id": "f003", "path": "阶段/SYS.2系统需求分析", "kind": "phase",
      "value": "系统需求分析", "source": "skeleton", "status": "guess",
      "authority": "PingCode汽车电子官方方案 + ASPICE SYS.2" }
  ],
  "pending_review": [
    { "raw": "他们说评审不在设计层做", "from": "meeting:2026-06-25", "why": "没匹配到明确 fact，待裁定" }
  ]
}
```

**字段**
- `id` — 稳定 id，manifest 用它把 fact 映射到 PingCode 对象 id。
- `path` — 挂在骨架哪个槽，**合并与父节点定位的匹配键**。
- `kind` — 渲染成哪种 PingCode 对象（work_item / phase / test / wiki / ticket / idea …，即 MVP 里**可渲染的对象类型**）。术语/命名修正**不**单列为 `term` 对象，而是落成对客户点名的那条对象 fact 的 `corrected` 改名（全局术语替换为后续，非 MVP）。
- `value` — 内容（皮）。
- `source` — 来源 provenance：`skeleton | doc:<file>#<loc> | meeting:<date>`。
- `status` — `guess | confirmed | corrected | rejected`。
- `authority` — 出处引用（仅骨架来源有，如 `PingCode汽车电子官方方案 + ASPICE SYS.2`），让"这结构凭什么"在元素层面可当场答。
- `pending_review` — 适配器判不准的输入，挂起待人裁，**不自动落**。

**状态语义**：`guess`=骨架放的草案（渲染但最弱）/ `confirmed`=文档或反应证实 / `corrected`=客户改了值 / `rejected`=客户否掉、渲染时删。

## 6. 合并优先级（唯一有脑子的逻辑，纯函数）

新补丁 P 撞上已有 fact E（按 `path` 匹配）时：

- **来源权威**：`meeting > doc > skeleton`；同级里**晚的盖早的**。
- **高权威赢 `value` 与 `status`**：
  - 骨架 `guess` + 文档 `confirmed(X)` → `confirmed`，值=X（**贴皮盖骨架**）。
  - 任意 + 会谈 `corrected(Y)` → `corrected`，值=Y。
  - 任意 + 会谈 `rejected` → `rejected`。
- **低权威永不能盖高权威**：重跑骨架**绝不能**把"会谈纠正过的"打回 `guess`。← 让 POC 一轮轮**收敛**而非翻烧饼。
- **匹配不上 → 进 `pending_review`**，不自动落。← 防误读一句反应就改错环境。
- 新 `path`（骨架无此槽，客户要的新结构）→ 追加为新 fact。

> 末两条（低不盖高、匹配不上挂起）是回路收敛与安全的命根，纯逻辑单测必须盖死。

## 7. 两个适配器（LLM 提补丁，绝不盲落）

- **Skin 适配器** `文档→补丁`：读 `.xlsx`（每行一个候选）/ `.docx`（语义段，复用 `pingcode-wbs`）。核心判断是"该条挂骨架哪个 `path`"——拿条目去和骨架 path 列表语义匹配；命中→`confirmed`/`source=doc` 补丁，**没把握→ `pending_review`**。
- **Reaction 适配器** `会谈→补丁`：读 `followup-extraction` 输出，每句客户话分类成 `confirm / correct(值) / reject / add`，按 `path` 找对应 fact；**判不准→ `pending_review`**。

> 两适配器都靠 LLM 语义判断，安全网同一张：**判不准就挂起 + 所有动作过 dry-run**，没有"自动直改客户环境"这条路。

## 8. Diff 渲染器（纯函数，无网络）

`(facts, manifest) → 计划{create[], update[], delete[]}`。按 `fact.id` 在 manifest 查建没建过，查表定动作：

| fact 状态 | manifest 里有? | 动作 |
|---|---|---|
| confirmed / corrected / guess | 没有 | **增**（按 kind 在 path 父节点下建） |
| corrected（值 ≠ 已建值） | 有 | **改**（改名/编辑） |
| rejected | 有 | **删** |
| rejected | 没有 | 空操作 |
| 没变 | 有 | 空操作 |

父节点定位：`path` 层级 → manifest 查父 id（复用现有 `parent_id`/`phase_id` 层级规则）。输出只是数据，可脱网测。

## 9. Applier + dry-run 护栏

- `--dry`（默认先走）：打印 `增 N / 改 M / 删 K`，每条写清"啥→啥"，**不碰网络**。
- 确认后落：`pingcode_api` 增/改/删，同级并发（复用 `batchCreateParallel`）；每成功一个 → 回写 manifest（`fact.id → obj.id`）+ 翻 fact 状态。
- **护栏**
  - **只删 manifest 里 fact 关联的对象**——绝不删我们没建的（护住客户手改/别处内容）；删目标不存在→跳过+告警。
  - **幂等**：无新补丁再跑 → 计划空（0/0/0）。
  - **部分失败**：标 fact `apply-failed`，可重跑（沿用 `build_manifest` 事务性）。
  - **已知局限**：信任 manifest——有人在两轮间手改环境会让它过时 → dry-run 预览把意外暴露出来，接受（未上方案 3）。

## 10. CLI 入口（一次调用 = 回路转一格）

```
node pingcode_revise.js --client_id=.. --client_secret=.. --identifier=JLALED \
  --context=某车灯Tier1_context.json \
  [--from-doc=reqs.xlsx | --from-followup=会谈0625.md] [--dry]
```

每来一份新文档 / 每见一次客户就跑一次：跑相应适配器产补丁 → 合并入 store → 渲染 → dry-run → （确认）落 + 回写。

## 11. 端到端时间线（验收即按此演）

| 时刻 | 输入 | store 变化 | 环境动作 |
|---|---|---|---|
| 0 刚搭完 | 骨架 | 全 `guess` | 像样但通用的 ASPICE 环境 |
| 1 客户给需求 Excel | `--from-doc` | 占位 `guess` + 文档 `confirmed("ADB分区点亮算法")` → `confirmed` | 占位**改名**+ 真需求**增** |
| 2 第一次见客户 | `--from-followup` | "评审独立门"→`rejected`；"叫矩阵大灯"→`corrected`；"少了功能安全线"→新 `confirmed`；模糊话→`pending_review` | **删** + **改名** + **增** + 挂起不动 |
| 2 后重跑骨架 | 骨架 | 骨架想把"矩阵大灯"打回占位 → 被"低不盖高"挡住 | 无（客户说过的话不被冲掉） |

## 12. 测试（沿用 `pingcode_tag_logic` / `vertical_template` 纯逻辑单测套路）

- **纯单测**（node:test，无网络）：§6 合并优先级（骨架+文档→confirmed、会谈纠正、**低不盖高**、匹配不上挂起）+ §8 渲染器决策表每种组合 + **幂等**（无变更→空计划）。
- **适配器**：样例 `reqs.xlsx` + 样例 `followup.md` → 期望补丁（golden）。
- **E2E 冒烟**（daocloud-test）：建小环境 → 喂文档（贴皮）→ 喂 mock 反应（改名+增+删）→ dry-run 计划对 → 落 → GET 校验 → 复跑幂等（0 操作）。

## 13. MVP 边界（首个实现计划范围）

- ✅ Context Store 形状 + provenance/状态字段
- ✅ 合并优先级纯逻辑（meeting>doc>skeleton、高盖低、低不盖高、匹配不上挂起）+ 单测
- ✅ Diff 渲染器纯逻辑（决策表 + 幂等）+ 单测
- ✅ Skin 适配器（`.xlsx` 必做，`.docx` 复用 wbs）
- ✅ Reaction 适配器（吃 `followup-extraction` 输出）
- ✅ Applier + dry-run 闸 + 护栏 + CLI
- ✅ E2E 冒烟（daocloud-test）
- ❌ 骨架生成（隔壁）、自动触发/loop 马达、跨客户驾驶舱、`.mpp` 解析

## 14. 未决 / 后续

- **首建如何初始化 context.json**：MVP 由首次搭建（Phase 3 蓝图 + Phase 4 manifest）落地后**导出**一份初始 store（全 `guess`/`skeleton`）。导出器是否并入本件 or 单列，待实现计划定。
- **`.docx`/官网贴皮的抽取质量**：MVP 先 `.xlsx`（结构最稳）跑通，`.docx` 复用 `pingcode-wbs` 起步，质量不足再迭代。
- **自动触发**：回路验证香了，再单独 brainstorming `/schedule` 无人值守马达（连同跨客户驾驶舱）。
- **`pending_review` 的人机交互形态**：MVP 先打印清单让用户口头裁定并回灌；是否做成交互式确认待定。
