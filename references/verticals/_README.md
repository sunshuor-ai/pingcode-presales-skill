# 行业结构模板（verticals）

每个 `<行业>.md` = 一个行业的完整结构包。搭建时按客户「行业」匹配（命中→加载 / 未命中长尾→现场调研生成 / 都不沾边→降级通用 WBS）。

## 文件格式
- **散文区**（人编辑 / LLM 读）：项目类型、阶段门流程、**工作项骨架**、测试维度、标准、术语、客户关注点。
- **工作项骨架（必写，因行业而异）**：该行业的层级分解用**哪些工作项类型**——不是所有行业都用敏捷 `史诗→特性→故事→任务`。builder 必须按此骨架建，而非写死敏捷。
  - 敏捷软件：`史诗 → 特性 → 用户故事 → 任务/缺陷`
  - 汽车电子(ASPICE) / 医疗器械(IEC62304)：`阶段 → 需求(系统/软件需求) → 子需求(架构/详细设计，需求型多级) → 任务(单元实现)`，双向追溯 需求↔测试用例（PingCode 原生支持「子需求」多级，`需求` 不收 `task` 子项但收 `子需求`）。
- **结构化区**：一个 ` ```yaml ` 块，`types:` 列出专属工作项类型与字段，供执行层精确建出。
- 文件**首行注释**写 synopsis（别名关键词），供 LLM 就近匹配：
  `<!-- synopsis: 汽车电子 / 车载 / ADAS / 智能座舱 / 主机厂 / Tier1 -->`

## YAML 约束（必须，否则校验不过）
- `group` ∈ `需求 / 任务 / 事务`
- 字段 `kind` ∈ 13 种 PingCode 属性类型：单行文本 / 多行文本 / 单选 / 多选 / 数字 / 日期 / 成员 / 级联单选 / 级联多选 / 进度 / 评分 / 链接 / 引用
- `单选 / 多选 / 级联单选 / 级联多选` 必须带 `options: [...]`
- 每个 type 必须有 `name`，每个 field 必须有 `name`

## 校验
- `node scripts/vertical_template.js`            # 校验全部模板
- `node scripts/vertical_template.js 汽车电子`    # 校验单个（可省 .md）

## 新增 / 沉淀一个行业
1. 复制现有 `.md` 改内容；或由搭建时长尾生成 → 交付时确认沉淀。
2. 跑校验通过。
3. 提交 GitHub，同事 `git pull` 即得。

## 专属类型怎么选（重要）
优先**补 PingCode 短板**：用行业标准对照 PingCode 能力，缺口处正是最该建的自定义类型
（如 ASPICE `SUP.10 变更请求` PingCode 不满足 → 建「变更请求」），既补功能又是演示讲合规的抓手。
