# PingCode 售前演示指南

## 官方推荐演示顺序

1. 基础页面布局 → 2. 工作台 → 3. 产品管理 → 4. 项目管理 → 5. 测试管理 → 6. 知识管理 → 7. 协作空间 → 8. 自动化

## Scrum 项目演示 10 步

1. 展示项目结构（Epic→Feature→Story→Task 层级树）
2. 打开 Product Backlog → 优先级排序
3. 进入当前 Sprint → 看板视图 + 故事点统计
4. 拖拽工作项 → 展示状态流转
5. 打开燃尽图 → 进度追踪（3种维度）
6. 点击工作项 → 展示关联的测试用例 + CI/CD 状态
7. 展示发布(Release) → 版本规划与交付进度
8. 打开工时统计 → 团队负载
9. 展示自动化规则 → 状态自动流转
10. 迭代回顾 → 持续改进闭环

## 产品管理演示 7 步（Ship 模块）

1. 展示产品库 → 多产品线管理
2. 打开 Web 站点渠道 → 客户自助提交工单
3. 工单清洗 → 分发至需求池
4. RICE 优先级评分 → 科学排序（工作量/价值/客户投票/触达/影响/信心）
5. 需求基线 → 版本快照
6. 需求→工作项转化 → 关联研发执行
7. 需求追溯矩阵(RTM) → 来源→执行全链路

## 客户常见问题应答

### 项目类型选什么？
> 推荐混合项目(hybrid)。不要的功能可以关，纯类型不能反向开启。除非你们做过 Scrum 培训且确认用标准 Scrum，否则一律推混�。

### Epic/Feature/Story 怎么用？
> 方案A（推荐）：Epic(月+)→Feature(两周)→User Story(3天-1周) 层层拆。方案B：不拆分，统一用 User Story。

### 工时要不要用？
> 工时非必用。登记会增加日常工作量。如果需要评估资源饱和度，工时+资源组件可以拖拽分配任务。

### 测试用例和缺陷的关系？
> 缺陷虽然在测试用例里产生，但缺陷的载体是项目。创建缺陷时务必选对项目和迭代。测试计划建议选"迭代"类型，跟研发迭代一一对应。

### 知识空间怎么组织？
> 两种维度：按产品线（每个产品一个空间）或按部门。页面支持插入工作项链接，写日报/数值可直接关联。

### 普通员工怎么快速找到自己的任务？
> 筛选器。创建个人视图，只显示"负责人=我"+"未完成"，省去在项目里一个一个找。这是执行层最欢迎的功能。

### 能对接第三方工具吗？
> 代码仓：GitLab/GitHub/Gitee + Jenkins（工作项详情页可看构建/部署状态）。通讯：企业微信/钉钉/飞书（消息推送+工作台使用）。登录：支持 SSO/SAML 2.0/OIDC + 企业微信/钉钉扫码。

### 私有部署安全吗？
> ISO 27001 + 等保三级。支持国密 SM4 加密。数据不出企业。私有部署需自备 SMTP 邮件服务和 AI 大模型 Key。

## 自动化场景示例

| 场景 | 触发器 | 动作 |
|------|--------|------|
| 新人入职 | 成员加入部门 | 自动加入所有项目/知识空间 |
| 任务完成联动 | 所有子任务完成 | 父工作项自动完成 |
| 高优缺陷响应 | 创建 P0 缺陷 | 邮件提醒+自动加关注人 |
| 外部反馈处理 | Webhook 创建工单 | 自动分类+分派负责人 |

## 演示注意

- 先展示"全家桶"全貌，再深入 2-3 个核心功能
- 用客户真实业务场景举例，不说通用术语
- 展示跨模块联动（工单→需求→工作项→测试用例→知识库）
- 136项操作权限 + 本地化配置是差异化亮点
- 筛选器是最被低估的功能——对执行层员工非常有吸引力
